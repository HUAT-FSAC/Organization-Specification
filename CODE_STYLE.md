# 代码风格规范

## 一、命名规范

### 1.1 变量命名
- **局部变量**：使用小驼峰命名法，如 `userName`
- **全局变量**：使用大驼峰命名法，如 `GlobalConfig`
- **常量**：使用全大写，单词间用下划线分隔，如 `MAX_LENGTH`
- **布尔变量**：使用 `is`、`has`、`can` 等前缀，如 `isActive`、`hasPermission`

### 1.2 函数命名
- **普通函数**：使用小驼峰命名法，如 `calculateTotal`
- **构造函数**：使用大驼峰命名法，如 `User`
- **私有函数**：使用下划线前缀，如 `_privateMethod`
- **回调函数**：使用 `on` 前缀，如 `onClick`

### 1.3 类命名
- 使用大驼峰命名法，如 `UserService`
- 类名应清晰表达其职责，如 `UserRepository`

### 1.4 文件名
- **JavaScript/TypeScript**：使用小驼峰命名法，如 `userService.js`
- **Python**：使用蛇形命名法，如 `user_service.py`
- **Java**：使用大驼峰命名法，如 `UserService.java`
- **CSS**：使用短横线分隔，如 `user-profile.css`

### 1.5 目录命名
- 使用小写，单词间用短横线分隔，如 `src/components`
- 避免使用特殊字符和空格

## 二、代码格式

### 2.1 缩进
- **JavaScript/TypeScript**：使用 2 个空格缩进
- **Python**：使用 4 个空格缩进
- **Java**：使用 4 个空格缩进
- **HTML/CSS**：使用 2 个空格缩进

### 2.2 换行
- 每行代码长度不超过 100 个字符
- 长表达式应换行，保持逻辑清晰
- 逗号后应换行，如对象和数组的多行定义

### 2.3 空格
- 操作符前后应加空格，如 `a + b` 而非 `a+b`
- 函数参数逗号后应加空格，如 `function(a, b)` 而非 `function(a,b)`
- 大括号前应加空格，如 `if (condition) {` 而非 `if (condition){`
- 注释符号后应加空格，如 `// 注释` 而非 `//注释`

### 2.4 大括号
- 使用 K&R 风格，即左大括号与语句同行，右大括号单独一行
- 如：
  ```javascript
  if (condition) {
    // 代码
  }
  ```

## 三、注释规范

### 3.1 单行注释
- 使用 `//` 开头
- 注释内容应简洁明了，说明代码的功能或逻辑
- 如：
  ```javascript
  // 计算用户总分
  const totalScore = calculateScore(user);
  ```

### 3.2 多行注释
- 使用 `/* */` 包裹
- 用于注释复杂的代码块或函数说明
- 如：
  ```javascript
  /*
   * 计算用户总分
   * @param {Object} user - 用户对象
   * @return {number} 总分
   */
  function calculateScore(user) {
    // 代码
  }
  ```

### 3.3 JSDoc 注释
- 用于函数、类、模块的文档注释
- 应包含参数、返回值、描述等信息
- 如：
  ```javascript
  /**
   * 计算用户总分
   * @param {Object} user - 用户对象
   * @param {number} user.score - 用户得分
   * @return {number} 总分
   */
  function calculateScore(user) {
    // 代码
  }
  ```

### 3.4 文件头部注释
- 每个文件应包含文件头部注释，说明文件的功能、作者、创建时间等
- 如：
  ```javascript
  /**
   * 用户服务模块
   * @author 作者名
   * @created 2024-01-01
   * @description 处理用户相关的业务逻辑
   */
  ```

## 四、代码结构

### 4.1 函数结构
- 函数应保持简短，单个函数不超过 50 行
- 函数应只做一件事，保持职责单一
- 函数参数不应超过 5 个，超过应使用对象

### 4.2 模块结构
- 每个模块应保持职责单一
- 模块间应通过明确的接口进行通信
- 避免模块间的紧耦合

### 4.3 目录结构
- 按功能模块划分目录
- 公共组件和工具应放在单独的目录中
- 测试文件应与被测试文件放在同一目录下，或放在 `__tests__` 目录中

## 五、语言特定规范

### 5.1 JavaScript/TypeScript
- 使用 `const` 和 `let`，避免使用 `var`
- 优先使用箭头函数，特别是回调函数
- 使用模板字符串，如 `` `Hello ${name}` ``
- 避免使用 `eval()` 和 `with` 语句
- 优先使用 `===` 和 `!==`，避免使用 `==` 和 `!=`

### 5.2 Python
- 遵循 PEP 8 规范
- 使用 `snake_case` 命名变量和函数
- 使用 `CamelCase` 命名类
- 避免使用单字符变量名，除非是循环变量

### 5.3 Java
- 遵循 Java 编码规范
- 使用 `CamelCase` 命名类和方法
- 使用 `snake_case` 命名常量
- 类成员变量应使用 `private` 修饰符，通过 getter/setter 访问

### 5.4 HTML/CSS
- HTML 标签应使用小写
- CSS 选择器应使用语义化的类名
- 避免使用内联样式
- 使用 CSS 预处理器（如 Sass、Less）时，应遵循其语法规范

## 六、代码质量

### 6.1 可读性
- 代码应易于理解和维护
- 避免使用过于复杂的表达式
- 变量和函数名应清晰表达其用途

### 6.2 可维护性
- 代码应具有良好的结构和组织
- 避免重复代码，应提取公共逻辑为函数
- 应使用适当的设计模式

### 6.3 性能
- 避免不必要的计算和操作
- 合理使用缓存
- 优化算法和数据结构

### 6.4 安全性
- 避免使用不安全的函数和方法
- 对用户输入进行严格验证
- 防止 SQL 注入、XSS 等安全问题

## 七、工具配置

### 7.1 ESLint
- 配置 ESLint 规则，确保代码风格一致
- 集成到编辑器和 CI/CD 流程中

### 7.2 Prettier
- 使用 Prettier 自动格式化代码
- 配置统一的格式化规则

### 7.3 编辑器配置
- 为团队统一编辑器配置，如 VS Code 的 `.vscode/settings.json`
- 确保缩进、换行等设置一致

## 八、规范执行

### 8.1 代码审查
- 在代码审查中检查代码风格是否符合规范
- 对不符合规范的代码提出修改建议

### 8.2 自动化检查
- 使用 CI/CD 工具自动检查代码风格
- 配置 pre-commit 钩子，在提交前检查代码风格

### 8.3 培训和学习
- 定期组织代码风格培训
- 分享代码风格最佳实践

---

## 结语

本代码风格规范旨在确保团队代码的一致性和可读性，提高代码质量和维护性。团队成员应严格遵守本规范，并在实践中不断完善和改进。
