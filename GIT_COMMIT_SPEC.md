# Git Commit 规范

## 一、提交信息格式

### 1.1 基本格式
- 提交信息应包含三个部分：标题、正文和页脚
- 格式如下：
  ```
  <type>(<scope>): <subject>
  
  <body>
  
  <footer>
  ```

### 1.2 标题
- 标题应简洁明了，不超过50个字符
- 标题应使用英文动词开头，如 `add`、`fix`、`update` 等
- 标题应使用小写字母，结尾不加句号

### 1.3 正文
- 正文应详细说明提交的内容和原因
- 每行不超过72个字符
- 应使用英文，保持语言简洁明了

### 1.4 页脚
- 页脚用于引用相关的issue或PR
- 如：`Closes #123` 或 `Related to #456`

## 二、提交类型

### 2.1 主要类型
- **feat**：添加新功能
- **fix**：修复bug
- **docs**：修改文档
- **style**：修改代码风格，不影响功能
- **refactor**：重构代码，不添加新功能或修复bug
- **test**：添加或修改测试
- **chore**：修改构建过程或辅助工具

### 2.2 类型说明
- **feat**：用于添加新功能的提交
- **fix**：用于修复bug的提交
- **docs**：用于修改文档的提交，如README、API文档等
- **style**：用于修改代码风格的提交，如缩进、空格、格式化等
- **refactor**：用于重构代码的提交，如重命名变量、提取函数等
- **test**：用于添加或修改测试的提交
- **chore**：用于修改构建过程或辅助工具的提交，如修改webpack配置、添加依赖等

## 三、作用域

### 3.1 作用域定义
- 作用域用于指定提交影响的范围
- 作用域应使用小写字母，如 `auth`、`user`、`api` 等
- 作用域应与项目的目录结构或模块名称一致

### 3.2 常见作用域
- **auth**：认证相关
- **user**：用户相关
- **api**：API相关
- **db**：数据库相关
- **ui**：用户界面相关
- **build**：构建相关
- **test**：测试相关

## 四、提交信息示例

### 4.1 功能添加
```
feat(auth): 添加用户登录功能

添加了用户登录接口和相关验证逻辑

Closes #123
```

### 4.2 Bug修复
```
fix(user): 修复用户信息更新失败的问题

修复了用户信息更新时的参数验证错误

Closes #456
```

### 4.3 文档修改
```
docs(api): 更新API文档

添加了新接口的文档说明

Related to #789
```

### 4.4 代码风格
```
style(ui): 格式化CSS代码

统一了CSS代码的缩进和空格
```

### 4.5 代码重构
```
refactor(user): 重构用户服务模块

提取了公共方法，优化了代码结构
```

### 4.6 测试添加
```
test(auth): 添加登录功能测试

添加了登录功能的单元测试和集成测试
```

### 4.7 构建修改
```
chore(build): 更新webpack配置

添加了新的构建插件，优化了构建速度
```

## 五、提交规范执行

### 5.1 提交前检查
- 提交前应检查代码是否符合代码风格规范
- 提交前应运行测试，确保代码功能正常
- 提交前应检查提交信息是否符合规范

### 5.2 自动化检查
- 使用 git hooks 自动检查提交信息格式
- 集成到 CI/CD 流程中，确保所有提交都符合规范

### 5.3 提交信息模板
- 为团队提供提交信息模板，确保提交信息格式一致
- 模板示例：
  ```
  # 提交类型(作用域): 提交主题
  
  # 提交描述
  # 详细说明提交的内容和原因
  
  # 相关issue
  # Closes #123
  ```

## 六、常见问题处理

### 6.1 提交信息过长
- 标题应保持简洁，不超过50个字符
- 正文应详细说明，但每行不超过72个字符
- 如提交信息过长，应考虑拆分为多个提交

### 6.2 提交类型错误
- 应根据提交的内容选择正确的提交类型
- 如不确定提交类型，可参考提交类型说明

### 6.3 作用域不明确
- 作用域应与项目的目录结构或模块名称一致
- 如作用域不明确，可省略作用域

## 七、工具推荐

### 7.1 commitizen
- 使用 commitizen 工具生成符合规范的提交信息
- 安装：`npm install -g commitizen`
- 使用：`git cz` 代替 `git commit`

### 7.2 commitlint
- 使用 commitlint 工具检查提交信息格式
- 安装：`npm install --save-dev @commitlint/cli @commitlint/config-conventional`
- 配置：创建 `.commitlintrc.js` 文件

### 7.3 husky
- 使用 husky 工具配置 git hooks
- 安装：`npm install --save-dev husky`
- 配置：在 `package.json` 中添加 husky 配置

## 八、最佳实践

### 8.1 提交频率
- 应保持提交频率适中，每个提交应对应一个完整的功能或修复
- 避免一次性提交大量代码，应拆分为多个小提交

### 8.2 提交信息质量
- 提交信息应清晰明了，说明提交的内容和原因
- 应使用英文，保持语言简洁专业
- 应避免使用模糊的词汇，如 "fix bug"、"update code" 等

### 8.3 分支管理
- 应使用不同的分支进行开发，如 feature 分支、fix 分支等
- 应定期合并主干分支，避免分支冲突

### 8.4 代码审查
- 提交前应进行自我审查，确保代码质量
- 应通过 Pull Request 进行团队代码审查

---

## 结语

本 Git Commit 规范旨在确保团队提交信息的一致性和可读性，提高代码审查的效率和项目的可维护性。团队成员应严格遵守本规范，并在实践中不断完善和改进。
