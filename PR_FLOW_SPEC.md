# PR 流程规范

## 一、PR 创建前准备

### 1.1 分支管理
- **功能开发**：从 `develop` 分支创建 `feature/xxx` 分支
- **Bug 修复**：从 `main` 分支创建 `hotfix/xxx` 分支
- **版本发布**：从 `develop` 分支创建 `release/xxx` 分支
- 分支命名应清晰明了，包含功能或修复的简要描述

### 1.2 代码准备
- 确保代码符合代码风格规范
- 运行测试，确保功能正常
- 进行自我代码审查，检查代码质量
- 确保提交信息符合 Git Commit 规范

### 1.3 相关文档
- 更新相关文档，如 API 文档、使用说明等
- 确保文档与代码同步

## 二、PR 创建流程

### 2.1 发起 PR
- 从功能分支或修复分支向目标分支发起 PR
- **功能分支**：目标分支通常为 `develop`
- **修复分支**：目标分支通常为 `main` 和 `develop`
- **发布分支**：目标分支通常为 `main`

### 2.2 PR 标题
- 标题应简洁明了，不超过 50 个字符
- 标题应包含 PR 的主要内容，如 "feat: 添加用户登录功能"
- 标题应使用英文，保持语言简洁专业

### 2.3 PR 描述
- 描述应详细说明 PR 的内容和原因
- 应包含以下内容：
  - 功能或修复的详细说明
  - 实现思路和方法
  - 测试情况
  - 相关的 issue 或 PR
  - 其他需要说明的事项

### 2.4 关联 Issue
- 关联相关的 issue，如 `Closes #123` 或 `Related to #456`
- 确保 issue 描述与 PR 内容一致

### 2.5 标签管理
- 添加适当的标签，如 `feature`、`bug`、`documentation` 等
- 标签应与 PR 的内容一致

## 三、PR 审查流程

### 3.1 审查者选择
- 选择熟悉相关代码的团队成员作为审查者
- 至少选择 2 名审查者，确保代码质量

### 3.2 审查内容
- **代码质量**：检查代码是否符合代码风格规范，是否存在潜在问题
- **功能实现**：检查功能是否符合需求，是否实现完整
- **测试覆盖**：检查测试是否充分，是否覆盖主要场景
- **文档更新**：检查相关文档是否已更新
- **性能影响**：检查代码是否存在性能问题
- **安全问题**：检查代码是否存在安全隐患

### 3.3 审查反馈
- 提供具体的反馈意见，说明问题和改进建议
- 使用 GitHub/GitLab 的评论功能，对具体代码行进行评论
- 反馈应客观、专业，避免个人情感
- 对于重大问题，应要求修改后重新提交

### 3.4 审查流程
1. 审查者收到 PR 通知后，应及时进行审查
2. 审查者在 PR 中添加评论，提出修改建议
3. 作者根据审查意见进行修改
4. 作者修改完成后，通知审查者重新审查
5. 审查者确认修改后，批准 PR

### 3.5 审查时间
- 审查者应在 24 小时内完成审查
- 对于复杂的 PR，可适当延长审查时间，但应及时通知作者

## 四、PR 合并流程

### 4.1 合并条件
- 所有审查者已批准
- 所有测试已通过
- 代码已符合代码风格规范
- 相关文档已更新

### 4.2 合并方式
- **功能分支**：使用 "Squash and merge" 方式，将多个提交合并为一个
- **修复分支**：使用 "Merge commit" 方式，保留完整的提交历史
- **发布分支**：使用 "Merge commit" 方式，保留完整的提交历史

### 4.3 合并后处理
- 合并后应删除已完成的功能分支或修复分支
- 通知团队成员 PR 已合并
- 更新项目文档，如版本更新日志等

## 五、PR 模板

### 5.1 模板内容
- **标题**：[类型] 功能或修复描述
- **描述**：
  - 功能或修复的详细说明
  - 实现思路和方法
  - 测试情况
  - 相关的 issue 或 PR
  - 其他需要说明的事项
- **检查列表**：
  - [ ] 代码符合代码风格规范
  - [ ] 运行测试，确保功能正常
  - [ ] 更新相关文档
  - [ ] 关联相关 issue

### 5.2 模板使用
- 在项目中创建 `.github/pull_request_template.md` 文件
- 发起 PR 时，系统会自动使用模板
- 作者应根据实际情况填写模板内容

## 六、常见问题处理

### 6.1 冲突处理
- 如 PR 与目标分支存在冲突，应先解决冲突
- 解决冲突后，重新提交代码
- 通知审查者重新审查

### 6.2 测试失败
- 如测试失败，应分析失败原因并修复
- 修复后重新运行测试，确保测试通过
- 通知审查者重新审查

### 6.3 审查意见处理
- 认真对待审查意见，虚心接受建议
- 对于有争议的问题，应与审查者进行沟通
- 确保所有审查意见都得到妥善处理

### 6.4 PR 积压
- 对于积压的 PR，应优先处理重要和紧急的 PR
- 定期清理积压的 PR，避免 PR 长时间未处理

## 七、最佳实践

### 7.1 PR 大小
- PR 应保持适当的大小，每个 PR 应对应一个完整的功能或修复
- 避免一次性提交大量代码，应拆分为多个小 PR

### 7.2 代码审查
- 审查者应认真审查代码，提供具体的反馈意见
- 作者应及时响应审查意见，进行必要的修改
- 团队应建立良好的代码审查文化，促进代码质量的提高

### 7.3 沟通协作
- 对于复杂的 PR，应提前与团队成员沟通
- 审查过程中，应保持良好的沟通，及时解决问题
- 鼓励团队成员积极参与代码审查，共同提高代码质量

### 7.4 持续改进
- 定期回顾 PR 流程，总结经验教训
- 根据实际情况，调整和优化 PR 流程
- 不断完善代码审查标准，提高代码质量

---

## 结语

本 PR 流程规范旨在确保团队协作的效率和代码质量的提高。团队成员应严格遵守本规范，并在实践中不断完善和改进。通过规范的 PR 流程，我们可以提高代码质量，减少问题发生，提升团队协作效率，最终交付高质量的项目成果。
